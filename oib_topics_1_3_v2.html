Тема 1 Методы защиты информации

1.1. Контроль целостности

Простыми словами
Как узнать, что данные не испортились при передаче? Добавить к ним «контрольный отпечаток» — и потом сверить. Если отпечаток не совпал — данные изменились.
Целостность информации — свойство данных: они не были изменены (случайно или намеренно) при хранении или передаче.

1.1.1. Контрольная сумма

Самый простой способ: сложить все байты данных. Результат — контрольная сумма. При получении пересчитываем и сравниваем.

Почему это слабо:
  • Не замечает перестановку байтов: A+B = B+A, сумма та же
  • Не замечает компенсирующие ошибки: один байт вырос на 1, другой упал на 1 — сумма та же
  • Легко подделать: можно изменить данные так, чтобы сумма не изменилась

1.1.2. CRC — Циклическая Избыточная Проверка

Простыми словами
CRC — это «умная контрольная сумма». Данные делятся на специальный «делитель» (полином), и остаток от деления и есть CRC. Любое изменение данных почти наверняка изменит остаток.
CRC (Cyclic Redundancy Check) — метод контроля целостности, основанный на делении данных как полинома. Гораздо надёжнее простого суммирования.

Полиномиальная арифметика в GF(2)

CRC работает с данными как с полиномами (многочленами) над полем GF(2) — полем из двух элементов {0, 1}.

Как читать
GF(2) — Galois Field of 2 elements. Это поле, где есть только 0 и 1.
Двоичное число представляется как полином: каждый бит — коэффициент при степени x.
1011 → \(x^3 + x + 1\) (биты 1,0,1,1 — коэффициенты при \(x^3, x^2, x, 1\))

Вся арифметика ведётся по модулю 2, то есть сложение = XOR, переносов нет.
Арифметика GF(2):
\(0 + 0 = 0,\quad 0 + 1 = 1,\quad 1 + 1 = 0\) (без переноса — это XOR)
Вычитание = сложение (т.к. \(-1 \equiv 1 \pmod{2}\))
Вывод: всё деление полиномов в CRC делается через XOR — никаких дробей.

Вычисление CRC — шаги

  1. Выбираем порождающий полином \(G(x)\) степени \(r\) — это фиксированная константа стандарта (например, CRC-32 имеет \(r=32\)).
  2. К сообщению дописываем \(r\) нулей справа (сдвигаем на \(r\) разрядов).
  3. Делим получившееся на \(G(x)\) — деление столбиком, но вместо вычитания используем XOR.
  4. Остаток от деления — это CRC (ровно \(r\) бит).
  5. Вместо нулей в конце сообщения ставим этот CRC. Передаём.

Проверка CRC

Получатель делит принятое сообщение на тот же \(G(x)\). Если остаток = 0 — всё нормально. Если нет — есть ошибка.

Пример CRC вручную
Сообщение: 1101, порождающий полином: 1011 (степень r = 3).

Шаг 1: дописываем 3 нуля → 1101 000
Шаг 2: делим 1101000 на 1011 через XOR:
1101000  ÷  1011
1011        (XOR первых 4 бит: 1101 XOR 1011 = 0110, сдвигаем)
────────
 1100 00    (опускаем следующий бит)
 1011       (XOR: 1100 XOR 1011 = 0111)
 ───────
  1110 0    (опускаем)
  1011      (XOR: 1110 XOR 1011 = 0101)
  ──────
   1010     (опускаем последний бит)
   1011     (XOR: 1010 XOR 1011 = 0001)
   ─────
    001  ← остаток (CRC)
Шаг 3: передаём 1101 001 (сообщение + CRC вместо нулей).
Проверка: получатель делит 1101001 на 1011 → остаток = 0 ✓
CRC применяется везде: Ethernet (CRC-32), USB, ZIP, PNG, протоколы передачи данных.
CRC гарантированно обнаруживает: все одиночные ошибки, все двойные ошибки, все ошибки нечётного числа бит, пакетные ошибки длиной ≤ r.

1.2. Защита в зашумлённых средах

Простыми словами
CRC только обнаруживает ошибки. Код Хемминга — исправляет их. Идея: добавить «контрольные биты» так, чтобы по их значениям вычислить, в какой позиции произошла ошибка.

Типы ошибок

  • Одиночная ошибка — перевернулся один бит.
  • Пакетная ошибка — повреждена последовательность смежных бит (помеха на канале).

Код Хемминга

Код Хемминга — линейный код, который исправляет одиночные ошибки и обнаруживает двойные.

Ключевое правило: контрольные биты ставятся на позиции 1, 2, 4, 8, 16, ... (степени двойки). Остальные позиции — данные.

Почему именно степени двойки?
Каждый контрольный бит отвечает за «свой» набор позиций — те, у которых в двоичном номере стоит 1 на соответствующем разряде:
— Бит на поз. 1 (=001₂) проверяет все позиции, где последний бит = 1: 1, 3, 5, 7 ...
— Бит на поз. 2 (=010₂) проверяет все позиции, где второй бит = 1: 2, 3, 6, 7 ...
— Бит на поз. 4 (=100₂) проверяет: 4, 5, 6, 7 ...
Тогда контрольные биты, которые «не сошлись» при проверке, составляют двоичное число — и это число равно номеру позиции с ошибкой.

Вычисление контрольных битов

Каждый контрольный бит выбирается так, чтобы XOR всех бит в его группе был равен 0.

Пример: кодируем 1011
Информационные биты: \(d_1=1, d_2=0, d_3=1, d_4=1\) (4 бита данных → нужно 3 контрольных бита → 7 позиций всего).

Раскладываем по позициям:
Позиция1234567
Типr₁r₂d₁r₃d₂d₃d₄
Значение??1?011

Вычисляем контрольные биты (XOR группы должен быть 0):
r₁ (поз.1): группа {1,3,5,7} → r₁ ⊕ 1 ⊕ 0 ⊕ 1 = 0 → r₁ = 0
r₂ (поз.2): группа {2,3,6,7} → r₂ ⊕ 1 ⊕ 1 ⊕ 1 = 0 → r₂ = 1
r₃ (поз.4): группа {4,5,6,7} → r₃ ⊕ 0 ⊕ 1 ⊕ 1 = 0 → r₃ = 0

Кодовое слово: 0 1 1 0 0 1 1

Исправление ошибки

  1. Получатель пересчитывает все контрольные биты заново.
  2. Если все сошлись — ошибок нет.
  3. Если нет — контрольные биты, которые «сломались», записываем как двоичное число (называется синдром — но важно только одно: это число = номер позиции с ошибкой).
  4. Инвертируем бит на этой позиции — готово.
Пример: ошибка в позиции 5
Передано: 0 1 1 0 0 1 1, получено: 0 1 1 0 1 1 1 (бит 5 перевернулся)

Пересчитываем:
s₁: {1,3,5,7} = 0 ⊕ 1 ⊕ 1 ⊕ 1 = 1 (не сошёлся)
s₂: {2,3,6,7} = 1 ⊕ 1 ⊕ 1 ⊕ 1 = 0 (сошёлся)
s₃: {4,5,6,7} = 0 ⊕ 1 ⊕ 1 ⊕ 1 = 1 (не сошёлся)

Синдром: s₃s₂s₁ = 101₂ = 5 → ошибка в позиции 5. Инвертируем 1→0. ✓

1.3. Эффективное кодирование (сжатие)

Простыми словами
Идея сжатия: частые символы кодируем короткими кодами, редкие — длинными. Итог: средняя длина кода меньше, чем фиксированные 8 бит/символ.
Средняя длина кода: \(\bar{L} = \sum p_i \cdot l_i\)
где \(p_i\) — вероятность символа, \(l_i\) — длина его кода

1.3.1. Код Фано (Шеннона-Фано)

Алгоритм простыми словами
1. Сортируем символы по убыванию вероятности.
2. Делим список на две половины с примерно равными суммами вероятностей.
3. Верхней половине дописываем «0», нижней «1».
4. Рекурсивно повторяем для каждой половины.
Пример: код Фано
СимволВероятностьДелим 1Делим 2Код
A0.40←«0»
(сумма=0.60)		«0»00
B0.20«1»01
C0.20←«1»
(сумма=0.40)		«0»10
D0.20«1»11
\(\bar{L} = 0.4 \cdot 2 + 0.2 \cdot 2 + 0.2 \cdot 2 + 0.2 \cdot 2 = 2.0\) бит/символ

1.3.2. Код Хаффмана

Алгоритм простыми словами
1. Берём два самых редких символа.
2. Объединяем их в один узел (сумма вероятностей).
3. Повторяем, пока не останется один корень.
4. По получившемуся дереву читаем коды: идём от корня, влево = «0», вправо = «1».
Хаффман гарантированно оптимален среди всех посимвольных кодов.
Пример: код Хаффмана — A(0.40), B(0.20), C(0.20), D(0.10), E(0.10)
Строим дерево снизу вверх:
Шаг 1: D(0.10) + E(0.10) → DE(0.20)
Шаг 2: B(0.20) + C(0.20) → BC(0.40) (берём два наименьших из оставшихся)
Шаг 3: DE(0.20) + BC(0.40) → BCDE(0.60)
Шаг 4: A(0.40) + BCDE(0.60) → корень(1.00)

Читаем коды по дереву:

\(\bar{L} = 0.4 \cdot 1 + 0.2 \cdot 3 + 0.2 \cdot 3 + 0.1 \cdot 3 + 0.1 \cdot 3 = 2.2\) бит/символ
Энтропия: \(H \approx 2.12\) бит/символ — код практически оптимален.

1.3.3. Арифметическое кодирование

Простыми словами
Вместо кода для каждого символа — одно число для всего сообщения. Сообщение кодируется как точка на отрезке [0, 1). Каждый следующий символ сужает этот отрезок.
  1. Каждому символу отводится часть отрезка [0, 1) пропорционально его вероятности.
  2. При чтении символов текущий отрезок последовательно сужается внутри подотрезка символа.
  3. Любое число из финального отрезка — это код всего сообщения.
Главное преимущество: может кодировать дробное число бит на символ (например, 0.3 бит/символ), что невозможно у Фано и Хаффмана (у них минимум 1 бит/символ). Поэтому арифметическое кодирование используется в JPEG2000, H.265.

1.3.4. Словарные алгоритмы: LZ77, LZ78, LZW

Простыми словами
Предыдущие алгоритмы требуют знать статистику символов заранее. LZ-алгоритмы строят словарь повторяющихся фраз прямо «на лету» — статистика не нужна.
Словарные алгоритмы — ищут повторы в данных, запоминают их в словарь, заменяют повторы ссылками на словарь.

LZ77 (1977)

  • Смотрит назад в уже просмотренные данные (скользящее окно = словарь).
  • Находит самое длинное совпадение с текущей позицией.
  • Выдаёт тройку: (смещение назад, длина совпадения, следующий новый символ).

LZ78

  • Строит явный словарь фраз (не скользящее окно).
  • Выдаёт пару: (индекс фразы из словаря, новый символ).
  • Каждая новая запись = найденная фраза + один новый символ.

LZW (Lempel-Ziv-Welch)

  • Улучшение LZ78: словарь сразу содержит все одиночные символы (в ASCII: коды 0–255).
  • Выдаёт только индекс (без нового символа — он неявно присоединяется к следующей фразе).
  • Применяется: GIF, TIFF, ранний ZIP.
Пример LZW: сжатие «ABABABA»
Начальный словарь: A=1, B=2

Тема 2 Методы стеганографии

Простыми словами
Криптография прячет содержание сообщения. Стеганография прячет сам факт его существования. Цель — чтобы противник вообще не догадался, что передача идёт.
Стеганография (от греч. «скрытое письмо») — сокрытие самого факта передачи информации путём встраивания её в безобидный объект (изображение, аудио, текст).

2.2. Стегоконтейнеры

Стегоконтейнер — файл-носитель, в который встраивается скрытое сообщение.

Виды контейнеров:

  • Пустой — файл до встраивания.
  • Заполненный (стего) — файл со встроенным сообщением.
  • Фиксированный — весь файл доступен заранее (изображение на диске).
  • Потоковый — данные поступают в реальном времени, встраивание «на лету» (аудиопоток, сеть).

Из чего состоит стегосистема:

  1. Сообщение M — то, что нужно скрыть.
  2. Контейнер C — файл-носитель.
  3. Стегоключ K — секрет, определяющий куда и как встраивать (например, порядок пикселей).
  4. Стегокодер: \(S = E_K(C,\, M)\) → заполненный контейнер S.
  5. Стегодекодер: \(M = D_K(S)\) → извлечение сообщения.

2.3. Стеганография в BMP — метод LSB

BMP — формат без сжатия. Каждый пиксель = 24 бита (по 8 бит на R, G, B).
Как работает LSB
В каждом байте цветового канала заменяем самый младший бит (LSB) на бит нашего секрета. Изменение значения на ±1 — человеческий глаз не замечает разницу между, например, красным 148 и красным 149.
  1. Берём байт пикселя (например, R=148 = 10010100).
  2. Заменяем последний бит на нужный бит сообщения.
  3. Записываем обратно.
  4. Повторяем для следующего байта и следующего бита сообщения.
Пример: встраиваем букву «A» = 01000001 в 8 пикселей (канал R)

2.4. Угрозы безопасности стегосистемы

УгрозаЧто делает противникКак защититься
Обнаружение (стегоанализ) Определяет, что контейнер содержит скрытое сообщение. Методы: статистический анализ LSB, визуальная атака, анализ гистограмм. Адаптивное встраивание (избегать однородных областей); рандомизация позиций через стегоключ; встраивание в частотную область вместо пространственной.
Извлечение Достаёт скрытое сообщение. Шифровать сообщение перед встраиванием + использовать стегоключ.
Уничтожение Разрушает скрытое сообщение (сжатие, изменение размера, фильтрация). Встраивать с помехоустойчивым кодированием; выбирать устойчивые к сжатию области (частотные коэффициенты).
Подмена Заменяет скрытое сообщение своим. Цифровая подпись + шифрование.

Тема 3 Арифметические базы криптозащиты

3.1. Модульная арифметика

Простыми словами
Модульная арифметика — это «арифметика по кругу». Как часы: после 12 идёт снова 1. Выражение a mod n — это остаток от деления a на n.
a mod n = r — остаток при делении a на n. Записывается: \(a \equiv r \pmod{n}\)
Пример: 17 mod 5 = 2, потому что 17 = 3·5 + 2.

Главное свойство — можно брать остаток на каждом шаге вычислений:

  • \((a + b) \mod n = ((a \mod n) + (b \mod n)) \mod n\)
  • \((a \cdot b) \mod n = ((a \mod n) \cdot (b \mod n)) \mod n\)
Шифр Цезаря через mod
Шифрование: \(E(x) = (x + k) \mod 26\)
Дешифрование: \(D(y) = (y - k) \mod 26\)
где x — номер буквы (A=0 ... Z=25), k — сдвиг.
Пример: буква D (=3), сдвиг k=3 → G (=6). Буква Z (=25), сдвиг k=3 → (25+3) mod 26 = 2 = C.

3.2. Взаимно простые числа и обратный элемент

Числа a и n взаимно просты, если у них нет общих делителей кроме 1: gcd(a, n) = 1.
Обратный элемент \(a^{-1} \pmod{n}\) — такое число b, что \(a \cdot b \equiv 1 \pmod{n}\).
Существует только если gcd(a, n) = 1.

Алгоритм Евклида — нахождение НОД

Принцип
НОД(a, b) = НОД(b, a mod b). Повторяем, пока второй аргумент не стал 0. Тогда первый = НОД.
Пример: gcd(252, 105)
gcd(252, 105) → 252 = 2·105 + 42  →  gcd(105, 42)
gcd(105, 42)  → 105 = 2·42  + 21  →  gcd(42, 21)
gcd(42,  21)  →  42 = 2·21  + 0   →  gcd(21, 0) = 21
Ответ: gcd(252, 105) = 21

Расширенный алгоритм Евклида — нахождение обратного элемента

Зачем нужен
Чтобы найти a⁻¹ mod n — это нужно, например, для дешифрования в RSA и аффинном шифре. Алгоритм находит x такое, что a·x ≡ 1 (mod n).
Пример: найти 7⁻¹ mod 26 — т.е. число x такое, что 7·x ≡ 1 (mod 26)
Прямой ход (алгоритм Евклида): 
26 = 3·7 + 5
 7 = 1·5 + 2
 5 = 2·2 + 1
 2 = 2·1 + 0   ← стоп, НОД = 1
Обратный ход (выражаем 1 через 7 и 26): 
1 = 5 - 2·2                       (из строки 3)
1 = 5 - 2·(7 - 1·5) = 3·5 - 2·7  (подставляем 2 = 7 - 5)
1 = 3·(26 - 3·7) - 2·7 = 3·26 - 11·7  (подставляем 5 = 26 - 3·7)
Итого: \(7 \cdot (-11) \equiv 1 \pmod{26}\)
Отрицательное: -11 mod 26 = 15
Ответ: 7⁻¹ ≡ 15 (mod 26)
Проверка: 7 · 15 = 105 = 4·26 + 1 ✓

3.3. Проверка числа на простоту

Зачем это нужно
Для RSA нужны большие простые числа (сотни цифр). Перебирать все делители невозможно — используют вероятностные тесты.
Тест Ферма (ненадёжный): если n простое, то для любого a выполняется \(a^{n-1} \equiv 1 \pmod{n}\). Обратная проверка: если это не выполняется — n точно составное. Но есть хитрые числа — числа Кармайкла (например, 561 = 3·11·17), которые проходят тест Ферма, хотя и не простые. Поэтому тест Ферма не используют в серьёзных задачах.

Тест Миллера-Рабина (надёжный)

Идея
Улучшение теста Ферма. Проверяем более строгое условие — числа Кармайкла его не проходят. Если число «сломалось» хотя бы раз — оно точно составное. Если прошло k раундов — скорее всего простое (ошибка < (1/4)ᵏ).
  1. Представляем n-1 как \(2^s \cdot d\), где d — нечётное.
    Как: делим n-1 на 2 столько раз, пока не получим нечётное число. Количество делений = s, остаток = d.
    Пример: n=13 → n-1=12 = 4·3 = 2²·3, значит s=2, d=3.
  2. Выбираем случайное a от 2 до n-2.
  3. Считаем \(x = a^d \mod n\).
  4. Если x=1 или x=n-1 — этот раунд пройден, идём к шагу 2 с новым a.
  5. Иначе повторяем s-1 раз: возводим x в квадрат по модулю n (x = x² mod n).
    Если на каком-то шаге x стало n-1 — раунд пройден.
    Если так и не получили n-1 — n точно составное, стоп.
Вероятность ошибки (принять составное за простое): ≤ 1/4 за один раунд.
За 40 раундов: вероятность ошибки ≈ (1/4)⁴⁰ — это меньше, чем вероятность аппаратного сбоя в компьютере.

3.4. Алгоритм RSA

Идея RSA
Перемножить два больших простых числа легко. Разложить произведение обратно на множители — практически невозможно (при больших числах). Именно на этом основана защита RSA.

Функция Эйлера

\(\varphi(n)\) = количество чисел от 1 до n, взаимно простых с n.
Для \(n = p \cdot q\) (два простых): \(\varphi(n) = (p-1)(q-1)\)

Генерация ключей RSA

  1. Берём два больших простых числа p и q.
  2. Вычисляем модуль: \(n = p \cdot q\)
  3. Вычисляем: \(\varphi(n) = (p-1)(q-1)\)
  4. Выбираем e: \(1 < e < \varphi(n)\) и \(\gcd(e,\, \varphi(n)) = 1\) — открытая экспонента.
  5. Находим d: \(d = e^{-1} \pmod{\varphi(n)}\) через расширенный Евклид — секретная экспонента.
Открытый ключ: (e, n) — публикуем.
Закрытый ключ: (d, n) — держим в секрете.
Шифрование (открытым ключом): \(c = m^e \mod n\)
Дешифрование (закрытым ключом): \(m = c^d \mod n\)
Пример RSA с малыми числами
1. p = 3, q = 11 → n = 33, φ(33) = 2·10 = 20
2. e = 7 (gcd(7, 20) = 1 ✓)
3. d = 7⁻¹ mod 20 = 3 (т.к. 7·3 = 21 ≡ 1 mod 20)

Шифруем m = 5:
c = 5⁷ mod 33 = 78125 mod 33 = 14

Дешифруем c = 14:
m = 14³ mod 33 = 2744 mod 33 = 5 ✓

Цифровая подпись RSA

Простыми словами
Обратный порядок: подписываем закрытым ключом, проверяем открытым. Только владелец закрытого ключа мог создать подпись.
Подписание: \(\sigma = h(m)^d \mod n\) (хеш сообщения, зашифрованный закрытым ключом)
Проверка: \(h(m) \stackrel{?}{=} \sigma^e \mod n\) (расшифровываем открытым ключом, сравниваем с хешем)

3.5. Система Меркле-Хеллмана (задача рюкзака)

Идея
Задача о рюкзаке (subset sum problem): дан набор чисел и целевая сумма S — найди подмножество, дающее сумму S. В общем случае это NP-трудная задача (нет полиномиального алгоритма, растёт экспоненциально). Но при специальном выборе чисел (сверхрастущая последовательность) — решается за линейное время. Меркле-Хеллман прячет лёгкую версию внутри трудной через модульную арифметику.
Сверхрастущая последовательность — каждый следующий элемент больше суммы всех предыдущих.
Пример: {1, 2, 4, 8, 16, 32} — каждый элемент = сумма всех предыдущих + 1.
Задача рюкзака для такой последовательности решается жадно: берём наибольший элемент ≤ остатку суммы, вычитаем, повторяем.

Схема Меркле-Хеллмана:

  1. Секретный ключ: сверхрастущая последовательность {b₁, ..., bₙ}, числа m (> Σbᵢ) и w (взаимно простое с m).
  2. Открытый ключ: {aᵢ = w·bᵢ mod m} — выглядит как случайная, «трудная» последовательность.
  3. Шифрование: сообщение x = (x₁, ..., xₙ) — двоичный вектор.
    Шифротекст: \(S = \sum x_i \cdot a_i\)
  4. Дешифрование: вычисляем \(S' = w^{-1} \cdot S \mod m\) — это «скрытая» сумма для {bᵢ}, решаем лёгкую задачу рюкзака жадным алгоритмом.
Важно: система Меркле-Хеллмана взломана Шамиром в 1984 году. Используется только в учебных целях.

3.6. Частотный криптоанализ

Простыми словами
Моноалфавитный шифр заменяет каждую букву на другую фиксированную букву. Но частоты букв в шифротексте совпадают с частотами в открытом тексте — это и есть уязвимость.
Частотный анализ — взлом шифра подстановки через статистику: самая частая буква шифротекста → скорее всего самая частая буква языка.

Частоты в языках:

  • Русский: О ≈ 11%, Е ≈ 8.5%, А ≈ 8%, И ≈ 7.4%
  • Английский: E ≈ 12.7%, T ≈ 9.1%, A ≈ 8.2%

Алгоритм взлома:

  1. Подсчитать частоты всех символов в шифротексте.
  2. Сопоставить самые частые символы с самыми частыми буквами языка.
  3. Проверить гипотезу на биграммах (частые пары: TH, HE, ER в английском).
  4. Постепенно восстановить весь ключ подстановки.
Пример (английский)
Шифротекст: «XYZZ XZ XKZZ...»
Частоты: Z=35%, X=25%, Y=15%
В английском: E≈13%, T≈9%
→ Z = E? X = T?
Проверяем биграммы: XZ = TE (частая пара ✓)
Постепенно раскрываем весь алфавит.
Защита от частотного анализа:
  • Полиалфавитные шифры (Виженер) — разные подстановки для разных позиций, частоты «размазываются».
  • Современные шифры (AES, ChaCha20) — статистических закономерностей нет вообще.
